# ③ 무제한 GET 페이징 — 보류 (설계 합의 필요)

> 과부하·데이터 손실 감사(2026-06-23)의 마지막 남은 🔴 항목. ①(PATCH 트랜잭션)·
> ②(updateService 일괄 PUT)·naver-sync 트랜잭션·Slack biz/ops 분리는 완료·배포됨(git 히스토리).

## 배경 / 문제

- `server/api/reservations.ts` GET: 전체 예약 + 결제내역 + **전체 ReservationHistory**를 매 로드마다
  통째 조회(날짜 필터·페이징 없음) → 클라 메모리 적재. 이력은 수정마다 누적되는데 전량 로드.
- `server/api/customers.ts` GET: 전체 고객 + **전체 CustomerPointHistory** 통째 로드(같은 패턴).
- 현재 규모(예약 66/이력 4/고객 63)는 작아 즉각 문제는 아니나 **무한 증가 구조**(어제 P2028 쓰기의 읽기 버전).

## 범위 / 결정사항

- 서버(날짜범위·페이징 도입, 이력 lazy 로드) + 클라(달력이 전체 `reservationMap`에 의존)를 **동시 개편**해야 함.
- 아키텍처 변경 + 회귀 위험 → 설계부터 합의 후 별도 일정. 현재 데이터 런웨이가 길어 급하지 않음.

## 연결

- ②(updateService 일괄 PUT)의 확장성 한계 — 클라가 전체 예약을 메모리에 들고 affected를 계산 — 와
  **같은 뿌리**. ③을 풀면 ②의 확장성도 함께 해결. (②의 1MB body·단일 tx 2N statement 한계도 이때 같이 검토)
