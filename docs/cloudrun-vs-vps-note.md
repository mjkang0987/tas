# Cloud Run vs VPS — 인프라 이전 판단 메모

> 인프라(컴퓨팅) 이전을 고민할 때 꺼내보는 판단 기준 메모.
> 작성 기준: 2026-07-16 · 현재 구조: Cloud Build → Cloud Run + Supabase(Free, 서울 ap-northeast-2)

---

## 0. TL;DR

- **지금은 그냥 Cloud Run 유지가 정답.** 현재 월 비용 ≈ **0원**(Cloud Run 프리 등급 + Supabase Free).
- VPS로 옮기면 **공짜였던 걸 월 3만원 고정비로 바꾸는 손해**. 비용 절감 목적이면 이전할 이유 없음.
- 이전을 고려할 신호는 "비용이 큰 것"만이 아니라 **① 비용이 확실히 크고 + ② 트래픽이 하루 종일 평평할 때** 둘 다 맞을 때.

---

## 1. 현재 구조

| 계층 | 사용 중 | 비용(현재) |
| --- | --- | --- |
| 컴퓨팅 | Next.js standalone 컨테이너 → **Cloud Run** (사용량 과금, scale-to-zero) | ≈ $0 |
| 빌드/배포 | main 머지 → **Cloud Build** 자동 | $0 (하루 120분 무료) |
| DB | **Supabase** PostgreSQL (Free, 서울 ap-northeast-2) | $0 |

- 앱 런타임 DB 연결: 트랜잭션 풀러(Supavisor, **6543**).
- 마이그레이션/seed: direct(session, **5432**).
- 배포 규약: **"마이그레이션 먼저(수동) → 코드 배포(자동) 나중"** (스키마 변경 시 500 방지).

---

## 2. 용어: 구글의 "VPS"는 Cloud Run이 아니라 Compute Engine

| 구글 제품 | 정체 |
| --- | --- |
| **Cloud Run** (현재 사용) | 서버리스 컨테이너 — VPS 아님 |
| **Compute Engine (GCE)** | 진짜 VPS(빈 리눅스 VM). Vultr·Hetzner·카페24와 같은 급 |

---

## 3. VPS 제공사 가격 비교 (2 vCPU / 4GB 기준, 2026-07)

> ⚠️ **제약: Supabase가 서울(ap-northeast-2)에 고정.** 앱↔DB 왕복 지연 때문에
> **VPS도 서울/근처여야 함.** 유럽 초저가 VPS(Hetzner)는 이 앱엔 부적합(DB가 멀어 느려짐).

| 제공사 | 월 요금 | 트래픽 | 한국(서울) 근접 | 이 앱 적합성 |
| --- | --- | --- | --- | --- |
| Hetzner (독/핀) | ~$4 | 20TB 포함 | ❌ | ❌ DB 멀어서 느림 |
| Contabo (일본) | ~$5~7 | 32TB 포함 | △ | △ |
| **Vultr 서울** | ~$20 | 3TB | ✅ | ✅ |
| DigitalOcean | $24 | 4TB | ❌ | ❌ |
| **GCE 서울(e2-medium)** | ~$24~28 + 트래픽별도 | 별도 과금 | ✅ | ✅(생태계 유지) |
| **AWS Lightsail 서울** | $24 | 포함 | ✅ | ✅(Supabase와 동일 AWS 서울망) |
| 카페24 / 네이버클라우드 | 국내가 | - | ✅ | ✅ |

**결론**: Supabase(서울) 유지 시 실질 후보 = **Vultr 서울 / GCE 서울 / Lightsail 서울 / 국내 업체**.
초저가 유럽 VPS의 가격 이점은 지연시간 때문에 무효.

---

## 4. DB(Supabase)는 그대로 써도 무관

- Supabase는 네트워크로 붙는 독립 매니지드 DB → 앱이 어디서 돌든 무관.
- VPS로 옮겨도 `DATABASE_URL`(6543) / `DIRECT_URL`(5432) 그대로, 마이그레이션 규약도 그대로.
- **단, DB가 서울이므로 VPS도 서울권**이어야 응답속도 정상(3절 참고).
- Supabase Free 제한(별도 이슈, 컴퓨팅 이전과 무관):
  - ~1주 미사용 시 프로젝트 일시정지 / DB 500MB / 월 대역폭 5GB / 프로젝트 2개.
  - 실서비스 트래픽이 늘면 **VPS가 아니라 Supabase Pro($25)** 를 먼저 검토.

## 5. 언제 이전하나 — 판단 기준

### 5-1. "Cloud Run 3만원 = VPS 3만원"이 동급이 아닌 이유
1. **용량 차이**: Cloud Run 3만원어치 트래픽을 3만원짜리 VPS 1대가 못 버틸 수 있음(오토스케일 없음).
2. **숨은 인건비**: VPS는 패치·백업·모니터링·배포 파이프라인을 직접 떠안음 → 금액만 같아선 이득 아님.

### 5-2. 비용이 오르면 순서대로
1. **청구 내역 분해** — Cloud Run 컴퓨팅 / egress / 기타 중 뭐가 돈 먹는지.
2. **Cloud Run 튜닝** — `min-instances=0`(놀 때 0으로), 메모리/CPU 적정화, `max-instances` 상한 캡.
   - 프리 구간에서 요금 새는 1순위 = `min-instances`가 1로 잡혀 항상 켜져 있는 것.
3. **약정 할인** — 트래픽이 꾸준하면 Cloud Run 커밋 할인(최대 ~30%↓).
4. **그래도 크면 VPS 비교.**

### 5-3. 실제 이전 신호 (둘 다 충족)
- **① 금액**: Cloud Run이 월 **5만~십몇만원 이상 꾸준히**.
- **② 트래픽 모양**: 하루 종일 **평평**(스파이크형 아님).

| 트래픽 모양 | 유리한 쪽 |
| --- | --- |
| 스파이크형(영업시간만 몰림, 밤엔 0) | ✅ Cloud Run(한가할 때 0으로 축소) |
| 평평형(하루 종일 꾸준) | ✅ VPS(어차피 계속 켜둘 거면 고정비가 쌈) |

> 미용실 예약앱은 보통 **영업시간 스파이크형** → Cloud Run이 계속 유리한 성격.

## 6. VPS 이전 시 작업 목록 (참고 — 지금은 불필요)

앱이 이미 Docker 컨테이너라 코드 수정이 아니라 "운영 환경 세팅"이 대부분.

| 작업 | 난이도 | 메모 |
| --- | --- | --- |
| VPS 개설 + 보안(SSH·방화벽·non-root) | ⭐⭐ | |
| Docker (+Compose) 설치 | ⭐ | |
| 리버스 프록시 + 자동 HTTPS (Caddy 등) | ⭐⭐ | Cloud Run이 공짜로 해주던 것 |
| 환경변수/시크릿 이전 | ⭐⭐ | |
| 컨테이너 실행 + 재시작 정책 | ⭐ | |
| **CI/CD 교체**(Cloud Build → GH Actions SSH 배포) | ⭐⭐⭐ | 손 제일 많이 감 |
| DB: **Supabase 유지**(권장) | ⭐ | 자가호스팅은 ⭐⭐⭐⭐ |
| DNS를 VPS IP로 | ⭐ | |
| 백업(DB 덤프 크론) | ⭐⭐⭐ | Supabase 자동백업 대체 |
| 모니터링/로그/알림 | ⭐⭐⭐ | |

## 7. 최종 결론

- **현 시점: 이전 불필요.** 프리 구간이라 지금 구조가 비용·운영 양쪽으로 최적.
- 나중에 요금이 튀면 → **원인(트래픽 vs 설정) 확인 → Cloud Run 튜닝/약정 → 그래도 크고 트래픽 평평하면 VPS**.
- VPS로 가더라도 **Supabase(서울) 유지 + VPS도 서울권**이 기본 전제.
